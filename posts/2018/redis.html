Redis 事务

星期四, 四月 12, 2018 on Redis, Python

Redis 的事务能够保证我们在事务中的一组命令能够按顺序地执行，执行期间 Redis 不会处理其它的客户端的命令请求

以下是 Redis 事务的简单用法:

redis> MULTI
OK

redis> SET book-name "Mastering C++ in 21 days"
QUEUED

redis> GET book-name
QUEUED

redis> SADD tag "C++" "Programming" "Mastering Series"
QUEUED

redis> SMEMBERS tag
QUEUED

redis> EXEC
1) OK
2) "Mastering C++ in 21 days"
3) (integer) 3
4) 1) "Mastering Series"
   2) "C++"
   3) "Programming"

在客户端发送 MULTI 命令后 Redis 会标记这个客户端（连接），并把这个客户端（连接）接下来发送的命令都放进一个 FIFO 队列，在接收到 EXEC 命令后 Redis 就会开始从队列中逐个取命令并执行。如果接收到的是 DISCARD，Redis 就会清空队列并把客户端（连接）标记回正常状态。

Redis 事务执行失败一般分两种情况：

  1. 在命令进入队列时，Redis 检测到命令的参数不合法，Redis 在接收到 EXEC 直接返回事务已被取消不再执行;

    redis> set a 1
OK

redis> multi
OK

redis> lpop a b
(error) ERR wrong number of arguments for 'lpop' command

redis> exec
(error) EXECABORT Transaction discarded because of previous errors.

  2. 在事务执行过程中，某命令操作不符合类型的 key，Redis 会继续执行剩下的命令；

    redis> set a 1
OK

redis> multi
OK

redis> lpop a
QUEUED

redis> exec
1) (error) WRONGTYPE Operation against a key holding the wrong kind of value

Redis 保证了事务中的命令要么全执行要么全不执行，所以 Redis 的事务操作是原子性的。

网上很多文章因为 Redis 不提供 roll back 就认为 Redis 的事务不是原子性操作，这些都是对原子性操作定义的误解。

不支持 roll back

在事务执行过程中产生的失败，Redis 并不提供 roll back 处理而是继续执行余下的命令，对于这样处理，Redis 作出了这样的解释：

  1. Redis 命令的执行失败只有在调用语法错误或者操作了不符合类型的 key 时出现，这些问题都应该在程序开发的期间被发现，并不会引入到生产环境中；
  2. Redis 很简洁很快，没必要增加 roll back。（roll back 实现复杂而且影响性能）

很多人可能对于第一点有异议，但第二点却十分具有说服力（KISS）。

WATCH

我们使用 WATCH 命令对一个或多个 key 进行监察，若被监察的 key 在事务执行前被修改，Redis 会把监察这个 key 的客户端（连接）标记，并不再它们接下来的事务操作，在接收到 EXEC 后返回 nil。

需要注意的是，使用 EXPIRES 操作被检察的 key 也被视为修改了 key，即使 key 还没过期。

redis> set a 1
OK

redis> watch a
OK

redis> expire a 60
(integer) 1

redis> multi
OK

redis> get a
QUEUED

redis> exec
(nil)
